Warum wir HTMX statt React für interne Tools verwenden

Drei interne Tools letztes Jahr, kein einziges Webpack-Update, kein einziges Bundle größer als 30 KB. Hier ist, warum HTMX in der Werkstatt funktioniert.

Was wir vorher hatten

Bis 2023 haben wir interne Tools — Backoffice für unsere Auftraggeber, Admin-Panels für unsere eigenen Apps — mit React+Next.js gebaut. Das funktionierte. Es funktionierte gut. Aber es kostete uns überproportional viel Wartung.

Jeder Webpack-Update brach etwas. Jede Next.js-Major-Version war ein halber Tag Migration. Bundle-Größen wuchsen organisch von 200 KB auf 800 KB, ohne dass irgendjemand benennen konnte, wann genau das passierte.

HTMX — die einfache Antwort

HTMX ist kein neues Framework, es ist eine Erinnerung daran, was HTML eigentlich kann. hx-get, hx-swap, hx-trigger — das war's. Es macht aus jedem Element ein potenzielles Form-Element, das HTML-Snippets vom Server holt und in die Seite einblendet.

Wir haben das erste Mal Anfang 2024 ein internes Tool damit gebaut — ein Admin-Panel für unser On-Call-System. Vorher: React + Next.js, Bundle 320 KB. Mit HTMX: Bundle 14 KB. Entwicklungszeit halbiert.

Drei produktive Tools, drei messbare Ergebnisse

  • On-Call-Admin (Vision GO intern). 14 KB Bundle, 1,2 s Time-to-Interactive, 4 Wochen Entwicklung. Alternative React-Schätzung: 8 Wochen.
  • Klinik-Backoffice (Kunde). 22 KB Bundle. Pflegekräfte loben das „schnelle” Tool — was 90 % der Reaktion auf Server-Roundtrips unter 80 ms ist.
  • SEPA-Reconciliation-Dashboard. Komplex auf den ersten Blick, aber im Kern ein Form-und-Tabelle-Tool. HTMX dauerte halb so lange wie der React-Vorgänger.

Wo HTMX nicht passt

Echtzeit-Co-Editing. Wenn Sie Notion oder Figma bauen, brauchen Sie CRDT-Sync und voll-clientseitige State-Management. HTMX kann das nicht.

Offline-First. PWAs mit Service-Worker und IndexedDB-Sync sind besser mit React Native oder Svelte zu bauen.

Mobile Apps. Niemand will eine HTMX-App im App Store. Native ist hier die Antwort.

Was HTMX bei uns ersetzt hat — und was nicht

HTMX ersetzt React in unseren internen Tools. Es ersetzt React nicht in unseren Endkunden-SaaS-Plattformen, wo komplexe Client-State-Synchronisation und Echtzeit-Updates Pflicht sind.

Die Regel, mit der wir mittlerweile arbeiten: Wenn das Tool keinen geteilten Client-State über Routen hinweg braucht, ist HTMX richtig. Wenn doch — bleiben wir bei React mit Server-Components.

Häufig gefragt

Ist HTMX ein React-Ersatz?

Nein. HTMX ersetzt React für Server-First-Apps mit minimaler Client-State. Für komplexe Single-Page-Apps mit globalem State bleibt React richtig.

Wie ist die Browser-Kompatibilität?

HTMX läuft in allen modernen Browsern (Chromium, Firefox, Safari). Keine Polyfills nötig für IE-freie Umgebungen.

Welches Backend funktioniert am besten mit HTMX?

Egal — Go, Python (FastAPI/Django), Ruby (Rails/Sinatra), Node (Express). HTMX ist Backend-agnostisch, solange Sie HTML-Fragmente zurückgeben können.
